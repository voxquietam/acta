# ADR 0002: Authentication

**Status:** accepted
**Date:** 2026-05-15

## Context

The team needs to log in to Acta. Options considered: standalone Django session auth (local users + passwords), SSO with `ksu24.back` (coupling to another service), Google OAuth (external identity provider).

Vox has no strong OAuth setup experience and no preference. Acta is intended to be **independent** of ksu24 to avoid creating coupling between services.

## Decision

**Google OAuth via `django-allauth`.** "Sign in with Google" is the only login path in MVP. No email/password registration.

## Why

- **No password storage** → no password resets, email verification flows, brute-force protection, or breach exposure to deal with.
- **`django-allauth`** is the standard Django package, supports 50+ providers, configured in `settings.py`, ships ready-made views and templates. ~30 minutes to set up.
- **The KSU24 team is already on Google Workspace** (university). One click = login.
- **Independent of ksu24:** if `ksu24.back` is down, Acta still works.
- **Not tied to ksu24 identity:** easy to invite outside collaborators who don't have a ksu24 account.

## Consequences

- Hard dependency on Google OAuth (if Google's API is unreachable, no one can log in). Acceptable for a self-hosted tool used by ~10 people.
- On first login, the user is created automatically (email + name from Google profile). Workspace membership is granted manually (see future `spec/permissions.md`).
- Local development: needs Google OAuth credentials for the dev environment. An optional fallback (`DEV_AUTH_BACKEND = 'django.contrib.auth.backends.ModelBackend'`) with test users via admin can be added — to be decided during setup.
- Adding SSO with ksu24 later is just another allauth provider; existing users are unaffected.

## Open Questions

- How `django-allauth` maps Google email → `User` — uniqueness rules to be confirmed at implementation time.
- Behavior if a user's Google email changes — edge case, deferred.

## Update — 2026-05-24: Google complements the invite/password flow

The original decision ("Sign in with Google is the only login path, no
email/password") was superseded as the product grew an invite-based
signup with username/password accounts created by admins. Google OAuth
is now an **additional** login path layered on top of that model:

- **Existing accounts** log in via Google when the provider's *verified*
  email matches a local account — they are signed in and the social
  account is linked automatically. No invite needed. This uses allauth's
  `SOCIALACCOUNT_EMAIL_AUTHENTICATION` /
  `SOCIALACCOUNT_EMAIL_AUTHENTICATION_AUTO_CONNECT` (both `True`). Safe
  because Google is a fully-trusted IdP for our org.
- **New accounts** can be created via Google only when an active
  `WorkspaceInvite` is in flight **and** the Google verified email
  equals the invite address. Otherwise social signup is refused —
  Acta stays invite-only. Enforced by
  `apps.accounts.adapters.NoSignupSocialAccountAdapter`.
- **Credentials** (Client ID / Secret from Google Cloud Console) are
  stored as a `SocialApp` row in Django admin, not in env/settings. The
  "Continue with Google" button only renders once that row exists. See
  `docs/operations.md` for the setup steps and the OAuth redirect URI.
