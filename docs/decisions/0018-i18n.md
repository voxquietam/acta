# ADR 0018: Internationalization and Localization

**Status:** accepted
**Date:** 2026-05-15

## Context

Acta is built primarily for the KSU24 team in Ukraine, but the codebase
itself is in English (see [0001](0001-stack.md) and the repo-level
``CLAUDE.md``). Users of the product expect a Ukrainian UI; future
contributors and any outside readers benefit from English. Adding i18n
support late is painful — every existing template string and every
admin label has to be retrofitted in one go. Doing it now is cheap
because there are very few translatable strings yet.

## Decisions

### Languages

Two languages in MVP, with explicit support to add more later without
schema migrations:

- **`en`** — default. Codebase source language; what's shown when no
  translation exists for a string.
- **`uk`** — Ukrainian. First-class translation target.

```python
LANGUAGES = [
    ("en", _("English")),
    ("uk", _("Ukrainian")),
]
LANGUAGE_CODE = "en"
```

Russian is deliberately not included. If a future external user
requests it, that's a separate decision.

### Per-user preference

Each user has a stored language preference: a ``language`` field on the
``accounts.User`` model. When set, it overrides cookie / Accept-Language
detection. When unset (new user), the resolver falls back to Django's
standard chain (cookie → Accept-Language → ``LANGUAGE_CODE``).

```python
class User(AbstractUser):
    language = models.CharField(
        max_length=8,
        blank=True,
        choices=settings.LANGUAGES,
        help_text="Preferred UI language. Overrides cookie and Accept-Language",
    )
```

### Middleware chain

```
SessionMiddleware
AuthenticationMiddleware
LocaleMiddleware                  # cookie / Accept-Language fallback
UserLanguageMiddleware            # overrides with User.language if set
```

``UserLanguageMiddleware`` (in ``apps/accounts/middleware.py``) runs
after Django's standard ``LocaleMiddleware`` so any user-specific
choice always wins.

### Switching language

A ``POST /accounts/set-language/`` endpoint accepts a ``language``
field, validates it is one of ``LANGUAGES``, saves it to
``request.user.language`` (for authenticated users) and to
``django_language`` cookie (for both authenticated and anonymous),
then redirects back to the previous page.

The UI surface is a dropdown in the topbar that submits to this
endpoint via a tiny HTML form (no JavaScript).

### What is translated

- **Template UI strings** — all visible labels, buttons, headings:
  ``{% load i18n %}`` + ``{% trans "..." %}`` / ``{% blocktrans %}``.
- **Admin sidebar labels** — already wrapped via ``gettext_lazy`` in
  ``UNFOLD`` config; translation lands in the same ``uk.po``.
- **Status / priority / health display names** — wrapped lazily as
  ``gettext_lazy`` constants in the model layer.

### What is NOT translated

- **Source code** (Python, comments, docstrings, ADRs): English only,
  per repo ``CLAUDE.md``.
- **`help_text` on model fields**: English source text. May be wrapped
  with ``gettext_lazy`` if and when help_text becomes visible to
  end users (currently shown only in Django admin tooltips).
- **User-generated content** (task titles, descriptions, comments,
  project update bodies): never translated; each user writes in their
  own language.
- **Internal enum keys** (`status="in-progress"`, `health="on_track"`):
  these are API identifiers and must not be localized.

### Tooling and workflow

- Catalog files live under ``locale/<lang>/LC_MESSAGES/django.po``.
- ``LOCALE_PATHS = [BASE_DIR / "locale"]``.
- Workflow:
  ```bash
  docker compose exec web python manage.py makemessages -l uk
  # …translators edit locale/uk/LC_MESSAGES/django.po…
  docker compose exec web python manage.py compilemessages
  ```
- ``compilemessages`` runs inside the container at deploy time so
  ``.mo`` artefacts are never committed. ``.po`` files are committed.

### Date / number formats

Django's locale-aware formatting (``USE_I18N = True``, ``USE_TZ = True``)
auto-localizes dates and numbers based on the active language. No
extra configuration needed for MVP.

### Translation source authority

The ``en`` text in templates and ``gettext_lazy`` calls is canonical.
Updating the English text invalidates the corresponding ``uk`` entry
(marked ``fuzzy`` by ``makemessages``); translators re-confirm.

## Why

- **English source + ``.po`` translations** is the universal Django
  pattern, well documented, and works with every translation tool
  (PyCharm, Weblate, Crowdin, Lokalise, plain text editors).
- **User-level preference** is the most predictable UX: pick once in
  settings, stays sticky across devices that log into the same
  account. Cookie-only would lose the choice on logout.
- **Default ``en`` + add ``uk``** keeps the codebase aligned with the
  repo language rule (everything committed in English) while giving
  the actual users a Ukrainian UI.
- **No Russian**: explicit decision, not an oversight; aligns with the
  user's context.

## Consequences

- Every new template needs ``{% load i18n %}`` at the top and
  ``{% trans %}`` around visible strings. Adding a string without
  wrapping it ships untranslatable UI.
- ``makemessages`` and ``compilemessages`` become routine commands;
  the Makefile (planned, see project memory) should expose them as
  ``make i18n-extract`` / ``make i18n-compile`` targets.
- ``locale/uk/LC_MESSAGES/django.po`` is generated by Django on first
  ``makemessages -l uk``; ``.po`` is committed, ``.mo`` is not.
- Verbose names on models (``Meta.verbose_name``) and choices display
  names will be added gradually as model labels start surfacing in the
  UI. Initial pass covers templates only.
- Switching language via cookie still works for unauthenticated users
  visiting the login page, which is desirable for a first-time UX.

## Open Questions

- Whether to add a translation hosting service (Weblate, Crowdin) once
  external contributors arrive. Deferred — start with hand-edited
  ``.po`` files and revisit when ``uk.po`` exceeds a few hundred
  strings.
- Whether to ship the date format as `dd.mm.yyyy` for ``uk`` (matches
  local expectation) — defer to first Ukrainian feedback.
