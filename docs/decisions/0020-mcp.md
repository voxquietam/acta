# ADR 0020: MCP Server (read + write tools, two transports)

**Status:** accepted
**Date:** 2026-05-19

## Context

Acta exposes a [Model Context Protocol](https://modelcontextprotocol.io/)
server so AI clients (Claude Desktop, Cursor, Cline, any MCP-aware
tool) can read and write the same data the web UI does — list / create /
update / archive / delete tasks, post comments, manage labels, query
activity for analytics. The MVP target is one human keeping their plan
in Acta while their LLM assistant operates on it without copy-paste.

Constraints:

- **Reuse domain logic, not bypass it.** Tools must go through the
  same validation, activity-log emission, and SSE broadcast that the
  web UI uses. No "MCP-only shortcuts" that would let an LLM put the
  DB into a state the UI can't reach.
- **Single-admin and multi-user reality.** The admin runs MCP from
  their laptop against the local dev DB; the team uses prod over
  HTTPS. One server, two transports.
- **Token, not session.** AI clients don't have a browser, so cookie
  auth is out. Each user mints API tokens in the web UI and pastes
  them into their MCP client config.
- **Live updates without bespoke wiring.** Every write surfaces in
  the existing SSE stream so the originating user's other surfaces
  (browser tab, other clients) stay in sync.

## Decisions

### Tool catalogue mirrors the verbs of the web UI

```
apps/mcp/
├── auth.py              # ApiToken handshake + per-token rate limiter
├── context.py           # IS_MCP_REQUEST ContextVar (see SSE section)
├── server.py            # build_server() — registers tools, dispatches
├── tools/
│   ├── __init__.py      # TOOLS = read.TOOLS + write.TOOLS
│   ├── _shared.py       # serialize_task_summary, resolve_*, FakeRequest
│   ├── read.py          # 7 read tools + N+1 guards
│   └── write.py         # 11 write tools (single + bulk)
├── views.py             # async Django view — HTTP transport
├── urls.py              # /mcp/
└── management/commands/mcp_serve.py   # stdio transport entrypoint
```

Each tool is a sync callable `(user, arguments) -> JSON payload`. The
unified `CALLABLES` dispatcher in `tools/__init__.py` lets the
transport layer stay transport-agnostic — both stdio and HTTP route
through the same handler. New tools get added by appending to
`read.TOOLS` / `write.TOOLS` and the same-named entry in
`read.CALLABLES` / `write.CALLABLES`; no transport wiring per tool.

### Two transports: HTTP for prod, stdio for local dev

| Transport | Audience | Setup |
|-----------|----------|-------|
| **HTTP** at `/mcp/` | All deployed users — paste URL + `Authorization: Token <secret>` into the client config | Single Django async view |
| **stdio** via `manage.py mcp_serve` | Local admin, dev-loop testing — client launches the server as a subprocess | `mcp.server.stdio_server` |

The HTTP path is the multi-user default — no SSH or wrapper scripts,
each user has their own token, revocation in `/accounts/settings/`
takes effect on the next call. The stdio path stays for local dev
because it never crosses the network and authenticates from
`ACTA_API_TOKEN` env, which is the safest way to test MCP work in
progress against a dev DB.

We deliberately skip the SDK's full Streamable HTTP transport
(`StreamableHTTPSessionManager`) — its session lifecycle + SSE
streaming buy nothing for short, sync tools. Every Acta tool resolves
in well under one HTTP timeout. If we ever add a streaming tool we
swap to the SDK transport behind the same URL.

### API tokens — hashed once, prefix kept for UI

`apps.accounts.models.ApiToken`:

- `token_hash` (SHA-256 of the secret) is the lookup key.
- `prefix` (first 8 chars of the plaintext) is stored to label rows
  in `/accounts/settings/` ("which one was that?").
- `ApiToken.generate(user, name)` returns `(instance, plaintext)` —
  the plaintext is shown **once** at creation and never persisted.
- `revoked_at` flips a soft-delete bit; the auth path checks it on
  every call.
- `last_used_at` is bumped via `update_fields=["last_used_at"]` to
  surface stale tokens without hot-pathing the row.

The same `ApiTokenAuthentication` DRF backend handles browser-less
API clients (curl, scripts, MCP). Registered **before**
`SessionAuthentication` in `DEFAULT_AUTHENTICATION_CLASSES` so it
short-circuits on the `Authorization` header without scanning cookies.

### Rate limit per token, per minute

`apps.mcp.auth.enforce_rate_limit` increments a Django-cache counter
keyed by `mcp:rl:<token_hash>:<minute_bucket>`. Default 60 req/min,
overridable via `ACTA_MCP_RATE_LIMIT_PER_MINUTE`. The counter blocks
runaway agent loops without throttling normal flows (~10–30 tool
calls per turn). LocMem cache for the single-worker dev setup, Redis
behind the same API for multi-worker prod if we ever go there.

### N+1 hygiene at MCP scope

Read tools call `_shared.user_workspace_ids(user)` once per request
and use the resulting list as `workspace_id__in=…`. Two queries
instead of a five-table JOIN through `workspace__memberships__user`,
no `.distinct()` pass, and the result fits Vox's "no N+1 — hard
rule" feedback. `test_read_tools.py` enforces this via
`CaptureQueriesContext` — adding rows must not move the query count.

### Writes use the existing serializers + activity log + SSE

Write tools route through `TaskSerializer` so validation matches the
web UI exactly: workspace membership, assignee-must-be-active-member,
subtask-depth-1. They call the existing `log_event()` /
`emit_task_diff_events()` paths so every change shows up in the
audit log with `actor = the authenticated user` and on the SSE
stream with pre-rendered card / row HTML.

One deliberate divergence from the UI: `acta_task_create` and
`acta_task_update` **auto-create missing labels** in the project's
workspace (deterministic hash-of-name colour, workspace-scoped
uniqueness preserved). Rationale: an LLM that hits "label not found"
gets stuck in a retry loop instead of producing the task it was
asked to. The web UI keeps strict validation because a human
operator can resolve the picker manually.

### `via_mcp` flag closes the SSE self-filter gap

`static/js/acta.js` drops SSE events whose `actor_id` equals the
current user — without that filter, every web-UI edit would
double-render (HTMX swap from the response + the SSE echo).

But when the *same* user writes through a *different* client
(Claude Desktop, Cursor, curl), their browser tab still needs the
SSE swap — the local tab has no other way to learn about the
change. We solve it with a contextvar:

- `apps.mcp.context.IS_MCP_REQUEST` flips to `True` inside both
  transport entrypoints (`views.mcp_http`, `server.call_tool`).
- `broadcast_task_events` and `log_event` stamp the payload with
  `via_mcp: true` when the flag is set.
- The client treats `via_mcp` as an override: apply the swap even
  for self-actor events.

ContextVars don't leak across concurrent ASGI requests (`asgiref`
propagates them through `sync_to_async`), so the flag stays scoped
to one tool call.

## Consequences

* The MCP server is a thin transport over the same domain code the
  web uses. New endpoints in DRF automatically have a corresponding
  MCP tool worth writing — and vice versa.
* Token revocation is instant: the next tool call hits a fresh `get`
  on `ApiToken`, sees `revoked_at`, and returns a clean auth error.
* Multi-user deployments don't need SSH per-user. Mint a token, paste
  into config, done.
* `via_mcp` is implementation-leaky in the SSE payload — any future
  client that wants to drop self-events but accept other-client
  writes from the same user has to know about the flag. That's
  acceptable because the alternative is a per-tab `client_session_id`
  cookie which adds three moving parts for the same outcome.
* Auto-create labels is a UX shortcut, not a security relaxation —
  the LLM still can't create labels in a workspace the calling user
  isn't a member of.
