## 2. Detailed findings by scope

### Modals (5 findings)

#### F1 — Modal root shell missing focus trap [P1]

**Where:** `templates/web/_modal_shell.html:26–49`

**What:** Modal backdrop + panel are rendered but neither focus trap nor focus restoration on close is implemented. The modal shell template has a `close()` method call on Esc + backdrop click, but:
- No `inert` attribute on sibling elements outside the modal to prevent focus escaping.
- No `aria-modal="true"` on the root (only on Cmd+K palette, not the shell).
- No save/restore of focus before and after modal visibility.

**Why it matters:** Keyboard-only users and screen-reader users can tab out of the modal to interact with the page behind it. On close, focus is not returned to the trigger button, breaking the user's context.

**Fix sketch:** Add `aria-modal="true"` to the `.acta-modal-backdrop` root (or create a wrapper with that role). Wire up a `FocusManager` in the Alpine x-data: on open, `this.previousFocus = document.activeElement` and apply `inert="true"` to body's direct children except `#modal-root`. On close, remove `inert` and `previousFocus.focus()`. Test with keyboard-only users.

**Effort:** 2–3 hours (Alpine lifecycle hooks, inert API browser support check, test coverage).

#### F2 — Task creation modal does not focus first input on open [P2]

**Where:** `templates/web/_create_task_modal.html:39`

**What:** Modal has `x-init="$nextTick(() => $refs.title?.focus())"` which correctly focuses the title input. However, this is a pattern that should be consistent across all modals. The shell template doesn't enforce or document this convention.

**Fix sketch:** Document in `_modal_shell.html` that child modals MUST implement focus management via `x-init` with a focus pattern. Create a reusable Alpine component or mention in CLAUDE.md that the pattern is `@acta:modal-opened.window="$nextTick(() => $refs.firstInput?.focus())"`. Audit other modals (workspace create, project create, bulk archive) for compliance.

**Effort:** 1–2 hours (documentation + 5 min per modal to verify).

#### F4 — Modal creation modal missing error messaging association [P1]

**Where:** `templates/web/_create_task_modal.html:82–244` (form fields)

### Dropdowns / Popovers (6 findings)

#### F6 — Status cell dropdown missing role="menu" and keyboard nav aria [P1]

**Where:** `templates/web/projects/_status_cell.html:31–95`

**What:** Dropdown is implemented as a positioned `<div>` with `x-show="open"` containing `<form>` + `<button>` pairs (line 63–91). No `role="menu"`, `role="menuitem"`, or keyboard navigation beyond click.

- Button group has no `role="menu"` wrapper.
- Each status option button has no `role="menuitem"`.
- No `aria-expanded` on the trigger button.
- No arrow-key navigation (user must click).

**Why it matters:** Screen-reader users won't understand the dropdown structure or be able to navigate with arrow keys (standard for menu patterns).

**Fix sketch:** Wrap dropdown in `<div role="menu" aria-label="Change status">`. Add `role="menuitem"` to each status button. On trigger button, add `:aria-expanded="open ? 'true' : 'false'"`. Add keyboard handlers: `@keydown.down.prevent`, `@keydown.up.prevent`, `@keydown.enter.prevent`, `@keydown.escape.prevent` for menu nav (similar to Cmd+K palette). Manage `data-focus-index` to track active item.

**Effort:** 2–3 hours (add roles + keyboard nav + test with screen reader + ensure focus visible).

### Images & Icons (3 findings)

#### F21 — Lucide SVG icons in templates lack aria-hidden or role="img" [P1]

**Where:** Throughout templates, `{% lucide "icon-name" "w-4 h-4" %}` renders inline SVG. Examples:
- `_task_row.html:28, 32, 49, 56, 64, 113`: status dot, priority, promote button, size, cycle, assignee.
- Base app topbar: multiple navigation icons.
- Modal shell close button: × icon.

**What:** Lucide macro renders `<svg>` elements. When used as decorative icons (status dots, navigation), they should have `aria-hidden="true"`. When used as meaningful images (e.g., an × that means "close"), they need `role="img"` + `aria-label`.

**Actual state:** Most icons are decorative (next to text that explains them). The × button in the modal shell has a visible label ("Close"), so aria-hidden is appropriate. **Most are correct by accident** (decorative + text label).

**Why it matters:** Redundant icon announcements (e.g., "chevrons-up" read aloud when the text already says "Urgent") clutter screen-reader output.

**Fix sketch:** Add `aria-hidden="true"` to the Lucide macro's SVG output by default. For icons that stand alone (status dot without text), add `role="img"` + `aria-label="Planned status"` to the icon span instead.

**Effort:** 1 hour (update Lucide macro + audit 10–15 icon uses).

### Colour Contrast (2 findings)

#### F24 — Placeholder text and hints may have low contrast [P2]

**Where:** Form labels and input placeholders throughout (`_create_task_modal.html`, filter sidebar, etc.).

**What:** Many labels use `text-placeholder-foreground` class. This is typically a muted gray (e.g., `text-zinc-500` or `text-gray-500`) on a light background. **No pixel-perfect measurement done**, but candidate classes:
- `text-placeholder-foreground`: likely ~4:1 or lower on `bg-card`.
- `text-subtle-foreground`: similar risk.

**Why it matters:** Users with low vision struggle to read faint labels and hints.

**Fix sketch:** Run WCAG contrast checker on actual rendered colors. If any are below 4.5:1 (AA standard), darken the foreground or lighten the background. For non-text UI elements (status dots, borders), 3:1 is acceptable.

**Effort:** 2–3 hours (screenshot + measure all text colors, propose palette changes, verify updated colors meet AA).

#### F25 — Status badge colors may not meet WCAG AA in dark mode [P2]

### Reduced Motion (2 findings)

#### F31 — Alpine x-transition animations ignore prefers-reduced-motion [P1]

**Where:** `_command_palette.html:60–69` (palette modal enter/exit transitions).

**What:** Palette uses `x-transition:enter="transition ease-out duration-120"` etc. No check for `prefers-reduced-motion: reduce`. If a user has enabled reduced motion in their OS (accessibility settings), the animation still plays.

**Why it matters:** Users with vestibular disorders or motion sensitivity experience discomfort from animations.

**Fix sketch:** Add Alpine effect or JS hook to check `window.matchMedia("(prefers-reduced-motion: reduce)").matches` on load. If true, skip or shorten transitions. Alternatively, use CSS media query:
```css
@media (prefers-reduced-motion: reduce) {
  [x-cloak] { animation: none !important; transition: none !important; }
}
```

But Alpine's `x-transition` is JS-driven, not CSS, so this won't work. Need JS approach: check on init + set `x-data` flag to conditionally apply transitions.

**Effort:** 2 hours (audit all x-transition uses, add prefers-reduced-motion check, test with OS motion setting disabled).

---

#### F32 — Tailwind transitions not respecting prefers-reduced-motion [P2]

**Where:** Throughout templates, inline `transition` class on buttons and UI elements (e.g., `_task_row.html:49`, `_assignee_avatar.html`).

**What:** Tailwind `transition` class applies CSS transitions without checking `prefers-reduced-motion`. 

**Actual state:** Tailwind does NOT auto-add prefers-reduced-motion support in v3 (it was added in v4). So this is a gap.

**Why it matters:** Users with motion sensitivity see rapid transitions throughout the UI.

**Fix sketch:** Add a global CSS rule or Tailwind plugin to respect prefers-reduced-motion. Add to `main.css`:
```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

Or configure Tailwind theme to disable transitions in reduced-motion context (requires custom config).

**Effort:** 1 hour (add CSS rule + test with OS motion setting).
