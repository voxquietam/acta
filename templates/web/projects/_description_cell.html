{% load i18n %} {% load static %} {% load web_extras %} {% comment %} WYSIWYG markdown description editor (TipTap, Linear-style). The user sees formatted text — never raw ``**bold**`` syntax — and selects text to surface the bubble menu with Bold / Italic / Code / Link / list / heading toggles. Wiring: * ``.description-editor-source`` is a hidden textarea holding the initial markdown source rendered server-side. The JS reads its value to seed the TipTap document. * ``.description-editor-mount`` is the element TipTap takes over. * ``.description-editor-bubble`` is the floating toolbar that ``BubbleMenu`` shows above the current selection. * ``input[name="description"]`` is kept in sync with the editor on every update so the form POST carries the latest markdown. * ``data-baseline`` stores the original markdown so the JS can skip saves when the value hasn't changed. Save fires on editor blur (``onBlur`` callback in ``description_editor.js``) via ``form.requestSubmit()`` which HTMX intercepts. A pagehide listener uses ``navigator.sendBeacon`` as a safety net for navigation-without-blur scenarios. {% endcomment %}
{% comment %} ``hx-swap="none"`` — the editor already holds the canonical text in the browser, so we don't ask the server for a fresh cell. The view returns only the OOB activity list which slots in via ``hx-swap-oob``. Re-rendering the cell would unmount TipTap, collapse the wrapper height for a frame, and visibly scroll the page. After a successful save the editor JS updates its baseline so a follow-up blur with the same content doesn't re-submit. {% endcomment %}
{% csrf_token %} {% comment %} Server-rendered fallback so the user sees the description immediately on page load — TipTap initializes as a JS module via ESM CDN, which has a measurable delay. The editor's mount script hides this fallback once the editor is ready. If JS fails to load entirely the fallback stays visible so the description is at least readable. {% endcomment %}
